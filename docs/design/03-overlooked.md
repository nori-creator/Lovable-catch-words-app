# 03. 見落としている重要事項

機能アイディアではなく、**放置するとローンチ後に致命傷になる**類のもの。優先度順。

## 1. セキュリティ: 共有 words テーブルの権限穴(即修正)

`words` は全ユーザー共有のキャッシュテーブル(同じ単語のカードを使い回す設計自体は正しい)だが、
現在の実装には穴がある:

- `updateWordExtras`(`src/lib/stickers.functions.ts:298`)が **word_id さえ知っていれば誰でも共有カードの解説・例文・読みを書き換えられる**。悪意がなくても、AさんがカードをカスタムするとBさんのカードも変わる。
- `saveSticker`(同ファイル)も、既存 word に対して extras を無条件で上書きしている(`:245` 付近)。

対策(推奨: copy-on-write):
1. `words` は「AI生成のマスター」として**ユーザーからは読み取り専用**にする(UPDATE権限のRLSを剥がす)
2. 個人のカスタマイズは `user_word_overrides (user_id, word_id, extras_patch jsonb, layout jsonb)` に差分保存し、表示時にマージ([04 §6](./04-architecture.md))
3. マスターの改善(明らかな誤りの修正)は「報告→再生成」フローに一本化する

あわせてやるべきRLS監査:
- `words_select_all` は匿名にも開いている(`USING (true)`)。単語データ自体は公開で問題ないが、意図的かどうかを確認して明文化する
- Storage の `stickers` バケットのポリシーは自己所有+可視性で守られているが、**posts で共有された画像**の署名URL寿命(現在6時間)と可視性変更時の失効を仕様化する
- Supabase の advisor(セキュリティ診断)を定期実行して確認する運用にする

## 2. プライバシー: このアプリは「顔+位置+行動履歴」を扱う

自撮り × GPS × タイムスタンプは、SNSの中でも最もセンシティブな組合せ。学習アプリのつもりで作っていると足をすくわれる。

原則(全機能に適用):
1. **デフォルト全部非公開**。共有は「この投稿を共有する」都度オプトイン(現状 stickers の visibility はデフォルト private になっており正しい。posts への引き上げ時が危険ポイント)
2. 共有・書き出し時は必ず: EXIF/GPSメタデータを除去、位置は「市区」レベルに丸める、自撮りを含めるかを毎回トグル表示
3. 友達タグ付け(構想中)は**タグされた側の承認制**。他人の顔が写った写真の公開は承認なしでは不可
4. 位置データの用途を「自分の記憶の索引」に限定し、他ユーザーの現在地・行動パターンが推測できる表示(リアルタイム地図など)は作らない
5. 未成年対策: 利用規約で13歳以上(または対象国の基準)を明記
6. プライバシーポリシーに「写真はAI解析のため外部AI APIに送信される」ことを明記(現在 Lovable AI Gateway → Google へ送信されている。これは法的表示義務レベルの話)

## 3. モデレーションと退会(ストア審査の必須要件)

App Store / Google Play は UGC(ユーザー生成コンテンツ)アプリに以下を要求する。**無いとリジェクトされる**:
1. コンテンツ通報機能(投稿・コメントに「通報」)
2. ユーザーブロック機能
3. 通報への対応フロー(まずは「通報されたら自動非表示→手動レビュー」で可)
4. **アカウント削除機能**(アプリ内から完結すること)。Supabase の user 削除+stickers/posts/storage のカスケード削除を実装
5. 利用規約・プライバシーポリシーへのリンク(routes に terms/privacy は存在するので、内容を実態に合わせて更新)

さらに推奨: データエクスポート(自分のカード・写真・日記をzipでダウンロード)。
「思い出データを人質に取らない」姿勢は口コミで効くし、GDPR系の要求にも先回りできる。

## 4. AIの嘘への防御(学習アプリの生命線)

間違った意味・読みを自信満々に覚えさせたら、このアプリの信頼は一発で終わる。現状は意味・注音・レベルが全てAI自己申告。

対策の段階設計:
1. **教育部辞書との突合(Now)**: 萌典(moedict)は台湾教育部の辞書データを持つオープンなプロジェクト。カード生成後に headword を照合し、
   - 一致 → カードに「✓ 辞書確認済み」バッジ、注音は辞書側を正とする
   - 不一致/未収録 → 「AI生成(未確認)」表示にし、ユーザーに手直し導線を出す
2. **TOCFL語彙表の静的テーブル化(Now)**: `words.level` をAIの推測でなく実際の語彙表ルックアップにする。これは図鑑コンプ率の分母([02 §5](./02-core-experience.md))と頻度表示([02 §6.4](./02-core-experience.md))の基盤にもなる。`vocab_lists (language, level, headword, rank)` テーブルを1回インポートするだけ
3. 例文・コロケーション・雑学は「AIの意見」であることをUI上で辞書項目と区別する(セクション見出しの色/アイコンを変える)
4. URLをAIに生成させない([02 §6.3](./02-core-experience.md))
5. `ai_runs` テーブル(既存)に生成物の検証結果も記録し、モデル変更時の品質比較に使う

## 5. AIコストの構造と防御

現状、1回の撮影で: 候補生成(vision)1回 + カード生成1回 + (復習のたびに)誤答生成2回×N + TTS毎回。
**コストの大半は復習時の誤答生成とTTSの重複生成**であり、[02 §3.2-3.3](./02-core-experience.md) のキャッシュ化で桁が変わる。

構造として持つべき防御:
1. `usage_events (user_id, kind, cost_estimate, created_at)` で **ユーザー別のAI使用量を記録**(kind: suggest/card/tts/correction…)
2. 無料枠の制限([05 §3](./05-roadmap-and-business.md)): 1日のcapture数・日記添削回数をこのテーブルで判定。superuser含め全員に適用されるサーバ側チェックにする(UIだけの制限は破られる)
3. words / tts / review_choices のキャッシュヒット率をダッシュボードで見られるようにする(ヒット率が上がるほど限界費用が下がる=スケールするほど安くなる構造を確認できる)
4. モデルは用途別に最小のものを選ぶ: 速報パスは小型vision、詳細パスは中型、添削は品質重視。プロバイダ抽象化([04 §3](./04-architecture.md))後にA/Bできるようにする

## 6. オフラインとレイテンシ

台湾のMRT・地下街・夜市の人混みは電波が不安定。「撮れない瞬間」が一番のペインの瞬間に起きる。

仕様: あとで解析キュー
1. 撮影時にネットワーク不通なら、画像を IndexedDB に保存して「📥 3枚が解析待ち」バッジをホームに表示
2. オンライン復帰(`navigator.onLine` + 再試行)で自動的に速報パス→詳細パスを順次実行
3. Service Worker([04 §4](./04-architecture.md))導入後は撮影ページ自体をオフラインで開けるようにする

レイテンシ予算(計測して守る数値):

| 区間 | 目標 |
|---|---|
| シャッター→速報(意味+音声) | 3秒 |
| 速報→詳細カード完成 | 15秒(バックグラウンド) |
| 復習タブ→1問目表示 | 1秒(事前生成が効いていれば達成可能) |
| 図鑑スクロール | 60fps(署名URLのN+1解消: `createSignedUrls` の一括API+ページネーション。`listMyStickers` は現在全件直列) |

## 7. 細かいが効く見落とし

- **単語の重複ステッカー**: 現在、同じ単語を撮り直すと sticker が複製され、DBトリガーで review も複製される(同じ単語に2つのSRS進捗ができる)。再遭遇ループ([02 §4](./02-core-experience.md))はこの整理を兼ねる
- **タイムゾーン**: journal/quests は Asia/Taipei 固定(`journal.functions.ts:35`)。多言語展開時は profiles にタイムゾーンを持たせる
- **端末間の写真画質**: アップロード前圧縮(長辺1600px程度+切り抜き用は768px)を標準化。ストレージコストと速報パスの速度の両方に効く
- **バックアップ心理**: 「アプリを消したら思い出が消える?」への回答をFAQに明記(クラウド保存済み+エクスポート機能)
- **通知許可の取り方**: 初回起動でいきなり求めない。「最初のカードを保存した直後」に「忘れる前に通知しようか?」と文脈つきで求める(許可率が段違い)
- **オンボーディング**: 目標は「インストール→60秒以内に最初のカード保存」。チュートリアルは作らず、最初のcaptureそのものをガイド化する(既存 onboarding ルートをこの思想で作り替える)
- **エラー時の体験**: AI失敗時(現在は toast エラー)に「手入力で続ける」フォールバックを必ず出す。せっかく撮った写真を無駄にしない
