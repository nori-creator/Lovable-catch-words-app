# Catchwords ローンチ・ロードマップ v1(docs/design/07-launch-roadmap.md)

> 目的: 「友達や彼女が普通に使える品質」→「ストア公開」→「収益化」への道筋を、優先順位つきで固定する。
> 本書は 06-scan-first-mvp.md(機能仕様)の上位にある**順序と品質の判断**を記録する。矛盾時は本書が優先。

---

## 0. 北極星と判断基準(変えない)

- 最大のペイン: **街で見たものの単語と発音が瞬間的に知りたい**。全ての優先順位はここから逆算する
- 機能追加の判断基準: 「この機能は**現実世界を教材にする体験**を強くするか?」強くしないなら、どれだけ便利でも入れない
- 品質の非交渉原則: 発音・意味の間違いは致命傷(検証済み/AI生成の区別を絶対に守る)。保存の摩擦を増やさない。スキャンに課金壁を置かない

## 1. フェーズ構成(この順番でやる。飛ばさない)

### Phase A: 「Wanrangテスト」合格 = 友達が普通に使える品質(目標: 2〜4週間)
ゴール: 説明ゼロでWanrongに渡して、初日に5枚キャッチし、翌日も自分から開くこと。

1. **Phase 4(復習リデザイン)とPhase 6(入力キャッチ/ゴースト再会/ワードツリー)の実装完了**(仕様書§8)
2. **オンボーディング実装**(本書§2の仕様)
3. **P0/P1バックログ消化**(本書§5)——特に写真表示速度
4. **実地計測**: 夜市・スーパー・MRTで各20スキャン。§9数値目標(ドット2.5秒/音声1秒/正答85%/辞書ヒット60%)を計測し、未達項目を潰す
5. **検証済み語のTTS事前一括生成**(仕様書§4.3。Claude Code+Batch APIで数百円)
6. **KPI計測の実装**(本書§3)
7. **βテスト**: Wanrong+友達2〜3人に配って1週間。毎日「今日開いた?なぜ?」だけ聞く

**合格基準**: β参加者の7日後継続 ≥ 50%、初回起動→初キャッチ到達率 ≥ 80%、クラッシュ・迷子ゼロ。

### Phase B: ストア公開の土木工事(目標: 1〜2ヶ月)
1. **Capacitor化**(Webアプリをネイティブアプリの皮で包む)。同時に解放されるもの: カメラの高速起動、プッシュ通知(SRS/場所リマインダーがついに本領発揮)、ホーム画面ウィジェット、manifest shortcutsのiOS版相当
2. **コスト・不正ガード**: 1ユーザー/日のスキャンソフト上限(例300回)、同一IP大量登録の検知、Supabase RLSの再監査
3. **法務・ストア要件**(チェックリスト):
   - プライバシーポリシーと利用規約のページ(位置情報・カメラ・写真の利用目的を明記)
   - アカウント削除機能(Apple必須要件)
   - App Store/Play Consoleのデータセーフティ申告(位置情報収集アプリは審査が厳しめ。用途説明文を丁寧に)
   - 権限リクエスト文言(なぜカメラ/位置が必要かをユーザー語で)
4. **Apple Developer($99/年)・Google Play($25買切)登録** → **TestFlight/内部テスト**でβ配布をストア経由に切替
5. 審査提出。**リジェクトは1〜2回あるのが普通**。慌てない

### Phase C: 収益化(ストア公開後)
1. 課金は**IAP(アプリ内課金)で実装**。Stripe計画は変更: アプリ内のサブスクはApple/Google決済が原則(手数料30%)
2. **必ずSmall Businessプログラム(Apple)/15%サービス手数料(Google)に登録** → 手数料15%に。NT$149の月額なら手取り約NT$127
3. リバーストライアル(登録後7日Pro)を実装。§11の無料枠(キャッチ3/日、AI添削3/日)を有効化
4. 価格はストア税込み表示で NT$150 / ¥700 帯に丸める
5. 収益の前提KPI: 無料→有料転換率のベンチマークは2〜5%。100人の友達ユーザーで2〜5人課金が現実的な出発点

## 2. オンボーディング仕様(最初の60秒)

- サインアップは**後回し方式**: 初回起動→すぐスキャン画面を体験(ゲスト)→初キャッチの保存時に初めて登録を求める(「この1枚を図鑑に残そう」)
- 起動直後: 3枚のスライド等は禁止。**1画面だけ**「かざして、タップしてみて」+カメラ許可のリクエスト(許可理由を1行:「見たものの単語を教えるため」)
- 初スキャンで検出ゼロだった時: 「文字や物にもう少し近づいて」+文字入力の逃げ道を必ず表示
- 初キャッチ完了時: 図鑑着地演出→「明日、この単語を覚えてるか聞くね」(SRSの予告=翌日の再訪理由を作る)
- 計測: 起動→初スキャン到達率、初スキャン→初キャッチ到達率をイベントとして記録

## 3. KPI(最小セット)

| 指標 | 定義 | 目標(β) |
|---|---|---|
| D1/D7リテンション | 初回起動の翌日/7日後に開いた割合 | 40% / 25% |
| 初キャッチ到達率 | 初回セッションでキャッチ1枚完了 | 80% |
| スキャン→キャッチ転換 | タップした単語のうち保存された率 | 15%以上 |
| 復習完了率 | 復習開始→フィードバック到達 | 70% |
| 辞書ヒット率 | スキャン結果の検証済みバッジ率 | 60% |

実装: scan_events/レビューログから日次集計するダッシュボードを管理ページに1画面追加(Lovableに依頼可)。

## 5. バックログ(2026-07-07受領分+判断)

| # | 項目 | 優先度 | 判断・送信用プロンプト |
|---|---|---|---|
| B1 | 写真表示が遅い(毎回読み込み) | **P0** | サムネイル未生成が原因の可能性大。→「アルバム/図鑑の画像表示が遅い。原因を調査して: サムネイル(小さい表示用画像)を保存時に生成しているか、signed URLを毎回再取得していないか、ブラウザキャッシュが効いているか。調査結果を報告してから、保存時サムネイル生成とキャッシュ活用で最速化して」 |
| B2 | 切り抜きが遅い | **P0** | まず無料の高速化: 「切り抜き処理に渡す画像を長辺1024px以下に縮小し、@imgly/background-removalの軽量モデル設定を使って速度を計測して。before/afterのミリ秒を報告」。外部API(有料/枚)への変更はこの結果を見てから判断 |
| B3 | 単語詳細に意味と例文しか出ない(詳しい解説・地図が出ない) | **P1** | 「図鑑のカード詳細シートに、生成済みカードの全項目(語法・コロケーション・類義語等)と、撮影場所のミニ地図を表示して。スキャンの詳細シートと図鑑の詳細で表示内容が同等になるように」 |
| B4 | 地図ピン→単語詳細へ飛ばない | **P1** | 「地図のピン(撮影画像ピン)をタップしたらその単語のカード詳細に遷移させて」 |
| B5 | ナビを5個に統合(中央=スキャン+カメラ統合) | **P1** | 「下部ナビを5個に再構成: ホーム/図鑑/[中央大ボタン: スキャン(かざす=調べる)+シャッター(撮る=キャッチ)を1画面に統合]/復習/設定。スキャン画面内にシャッターを置き、カメラ系の入口を1つにする」 |
| B6 | 日記: AIお手本廃止→添削+ネイティブの自然なフレーズ/チャンク/語法解説に変更 | **P1**(仕様変更) | 仕様書§6の日記関連を上書き: お手本全文生成はしない。ユーザーの文への添削+「その気持ちをネイティブならこう言う」フレーズ2〜3個+使った型の解説のみ |
| B7 | SRS忘却曲線の可視化(各単語の位置/分布) | P2 | Phase 6のワードツリーと同時に。「復習タブに忘却曲線グラフ: 横軸=時間、縦軸=記憶強度。各単語を点として配置し、危険ゾーンの単語数を表示。単語タップで詳細へ」 |
| B8 | CAPWORDS的な美しい切り抜き演出 | P2 | Phase 5(磨き込み)で。デザインはLovable委任の原則を維持 |

P0=今週、P1=Phase A中、P2=Phase A後半〜B。
