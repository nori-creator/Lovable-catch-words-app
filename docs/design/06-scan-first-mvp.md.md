# Catchwords 統合マスター仕様書 v2 — Scan-First MVP

> この文書は Catchwords の**唯一の正式仕様(Source of Truth)**である。
> リポジトリ内の `docs/design/01〜05` と矛盾する場合は**本書を優先**する。
> 保存先: `docs/design/06-scan-first-mvp.md`(Lovableにこのファイルを追加させる)。

## 0. この文書の使い方

- Lovable には **1フェーズ=1メッセージ** で送る(§8 のプロンプトをそのままコピペ)。
- 一度に全部送らない。各フェーズの「受入基準」を自分で確認してから次へ進む。
- 実装の微調整・バグ修正は Claude Code 側で行う(DBの変更は必ず `supabase/migrations/` に残し、両環境の同期を保つ)。
- **デザイン委任の原則**: この文書は機能・フロー・数値目標のみを定義する。視覚デザイン(色・形・アニメーション・質感)は本書では規定せず、Lovableのデザイン判断に委ねる。例外は機能的な意味を持つ指定のみ(例: 再会ドットの金色、仮画像と実写真の区別)。

## 1. 確定コンセプト

**「かざす=調べる、撮る=残す」**

最大のペインは「街で見たものの単語と発音が**瞬間的に**知りたい」。アプリの正体は**撮るだけで済む辞書**であり、図鑑・SRS・アルバムは「調べた結果が勝手に自分の資産になる」副産物である。

- **スキャン(かざす)**: 無摩擦・無制限。カメラをかざしてスキャン→映っているモノにドットが付く→タップ→単語+注音/拼音+日本語+ネイティブ発音が**即座に**表示・再生される。保存操作は不要。
- **キャッチ(撮る)**: 意志を持った保存の儀式。シャッター=キャッチ。切り抜きが図鑑にドンと着地する報酬演出。自撮り・一言感想は任意、場所・日時は自動記録。
- **復習**: 4択ではなく「自分の撮った写真を見て、その時の経験をその単語を使って話す」スピーキングoutput型。

### ターゲット(MVP)
台湾在住の日本人(繁体字・台湾華語・注音)。多言語対応は構造だけ残し、UIと辞書は zh-TW × ja に集中する。

## 2. MVPスコープ

### やる
1. スキャンモード(ドットUI、タップ即時表示、候補チップ、OCR優先)
2. 3層ハイブリッド辞書(検証済みDB → AI生成フォールバック → 出典ラベル)
3. タップ時の詳細プリフェッチ
4. キャッチフロー(切り抜き、図鑑着地演出、自撮り任意、一言感想)
5. スピーキングoutput型復習(音声デフォルト、ヒントボタン、AI添削+型の提示)
6. スキャンログ(調べたが保存しなかった単語の蓄積)
7. TTS音声のキャッシュと検証済み語の事前音声
8. 入力キャッチ(文字・音声=復唱による耳キャッチ)とフレーズカード(§5.2)
9. ゴーストカードと再会演出(§5.3)
10. ワードツリー(復習ごとに枝が1本ずつ解禁・蓄積される派生語設計、§6)

### やらない(将来フェーズ)
- ❌ 連続リアルタイムスキャン(ARグラス構想) — 単発スキャンのみ
- ❌ 発音チェックを保存の関門にする — 保存は撮影だけで完了
- ❌ 学習カリキュラム機能 — 型・チャンクは復習の中で提示する
- ❌ SNSフィード・友達機能の拡張 — 既存コードは温存するがMVPでは前面に出さない(§10 未決)
- ❌ 動画対応、SNS/記事からのタップ追加、企業コラボ

## 3. コア体験仕様: スキャンモード

### 3.1 画面と操作フロー

```
[カメラプレビュー]
  └ 下部中央: 大きなスキャンボタン
  └ 押す → その1フレームを解析(連続解析はしない)
  └ 解析中: 0.5〜2秒。待ち時間を楽しくする短い演出を入れる(演出内容はデザイン側で設計)
[結果オーバーレイ]
  └ 検出された各モノ/文字の位置にドット(モノと文字は見分けられるように区別)
  └ ドットをタップ →
      ・単語チップ表示: 「芒果  ㄇㄤˊ ㄍㄨㄛˇ · mángguǒ  | マンゴー」
      ・ネイティブ音声を自動再生
      ・チップ内に小さく「詳しく →」「+細かく」「📷キャッチ」
  └ 画面下部にタップした単語の要約ミニカード
```

- 表示するのは**単語・注音・拼音・日本語訳・品詞・出典バッジ**だけ。例文や解説はこの画面に出さない(速度最優先)。
- 検出ゼロの場合: 「文字入力で調べる」フォールバックを提示。
- スキャンは**ログインユーザーなら無制限**(コスト制御は使用量テーブルで監視のみ)。キャッチは無料枠1日n枚の既存制限に従う。

### 3.1b ドットの状態システム(発見レーダー)

検出された各単語を、ユーザーの学習履歴と照合して4つの状態で表示する。状態ごとに視覚的に明確に区別すること(具体的な色・光り方はデザイン側で設計。ただし「再会」は金色を基調とする)。

| 状態 | 判定条件 | 表示の意図 |
|---|---|---|
| **新発見** | ユーザーのカードにも過去のタップ履歴(scan_events.tapped=true)にも存在しない | 光って誘う。「まだ知らない単語がここにある」というワクワク |
| **再会** | ゴーストカード(写真なしカード)と一致 | **金色**に光る。「前に調べたやつだ!」の最高の瞬間 |
| **取得済み** | 写真ありカードと一致 | 控えめ表示+チェック。既に自分のものである安心感 |
| **通常** | 過去にタップしたが保存しなかった単語 | 標準のドット |

- 判定はスキャン結果が返った直後、クライアント側でユーザーのカード一覧・タップ履歴と照合する(追加のAI呼び出しなし)
- この照合のため、スキャン画面はユーザーの全カードのheadword一覧を起動時に軽量キャッシュしておく

### 3.2 AI検出コールの仕様

1回のスキャン = **1回のAI呼び出し**。物体検出とOCRを同じコールで行う。

**入力**: クライアント側で長辺1024pxに縮小・JPEG圧縮した画像(送信サイズ目標 300KB以下)。

**出力(JSONのみ、これ以外のテキスト禁止)**:
```json
{
  "items": [
    {
      "kind": "object",            // "object" | "text"
      "headword": "芒果",          // 台湾教育部準拠の繁体字。kind=textなら写っている文字そのまま
      "zhuyin": "ㄇㄤˊ ㄍㄨㄛˇ",
      "pinyin": "mángguǒ",
      "meaning_ja": "マンゴー",
      "pos": "名詞",
      "point": [512, 340],         // 正規化座標(0〜1000)。ドットを置く中心点
      "confidence": 0.93,          // 0〜1。AIの確信度
      "alternatives": ["木瓜"]     // confidence < 0.75 のときだけ、紛らわしい候補を1〜2個
    }
  ]
}
```

- 用語: **正規化座標** = 画像サイズに関係なく 0〜1000 の目盛りで位置を表す方式。表示時に実画面サイズへ掛け算で変換する。
- 検出数の上限: 8個(画面がゴチャつかない、トークン節約)。大きく写っているもの・学習価値の高いものを優先。
- プロンプトに含める制約: 台湾で実際に使う語(大陸語彙禁止)、TOCFL目標レベル以下優先、`kind=text` は**推測せず写っている文字をそのまま**返す。

### 3.3 タップ時の動作(即時表示+プリフェッチ)

タップした瞬間に並行して3つ走らせる:
1. **表示**: 検出結果に含まれる単語情報をチップ表示(追加のAI呼び出しなし=0秒)
2. **音声**: 辞書DBに音声があれば即再生。なければTTS生成(キャッシュ確認→なければ生成→Storageに保存)
3. **プリフェッチ**: `generateCard`(詳細カード生成)をバックグラウンドで開始し、結果をスキャンセッション内にキャッシュ。「詳しく→」を開いた時には出来上がっている状態にする。**タップされていない物体の詳細は絶対に生成しない**(コスト10倍防止)

### 3.4 OCR優先ルール

`kind=text` のアイテム(看板・メニュー・商品ラベル)は物体推測より信頼できるため:
- ドットの色を変えて区別(マゼンタ)
- 辞書DB照合はまず**完全一致**、なければ分かち書きして最長一致
- 多字数の文(店名スローガン等)は主要語だけドット化

### 3.5 「+細かく」階層ドット

タップチップ内の「+細かく」を押すと、**その物体の周辺領域を切り出して2回目の検出コール**を投げ、部分名称のサブドットを追加表示する(手→拇指/手掌/指甲)。MVPではオンデマンドの2回目コールで実装(初回コールに部分まで含めると遅くなるため)。

### 3.6 低信頼度時の候補チップ

`confidence < 0.75` のドットをタップした場合、断定表示せず「芒果?  木瓜?」の横並びチップを出し、ユーザーが選んだものを表示・記録する。誤った断定は学習アプリで最も信頼を失う。

### 3.7 スキャンログ(サイレント資産)

新テーブル `scan_events`: `user_id, headword, meaning_ja, kind, confidence, tapped(bool), caught(bool), lat, lng, created_at`。
- スキャン結果は全件、タップ・キャッチの有無つきで記録
- 「調べたけど保存しなかった単語」は後日「先週スキャンしたこれ、キャッチし忘れてない?」の再浮上通知に使う(通知実装は将来、テーブルだけ先に)
- プライバシー: 位置情報は本人のみ閲覧(RLSで保護)、共有機能には一切流さない

## 4. データ戦略: 3層ハイブリッド辞書

### 4.1 `dictionary_entries` テーブル(検証済みゾーン)

```
headword (text, PK相当・unique)   繁体字
zhuyin (text)                     注音
pinyin (text)
meaning_ja (text)
pos (text)                        品詞
tocfl_level (int, nullable)       1〜6
taiwan_usage (text)               'common' | 'written' | 'spoken' | 'rare'
audio_path (text, nullable)       Supabase Storage内の事前生成音声
source (text)                     'verified'(人手/公式データ確認済み) | 'ai'
entry_type (text)                 'word' | 'phrase'
scene_tags (text[], nullable)     ['night_market','mrt','supermarket',...] 将来のシーン図鑑用
notes (text, nullable)
```

- 初期データは**TOCFL華語八千詞のレベル1〜3(約1,500語)**を基に別途作成し、CSVで一括投入する(データ作成はこの仕様書の範囲外。投入用の管理画面 or SQLシードだけ用意)
- 教育部《國語辭典簡編本》は**照合・検証の参照元**として使う(CC BY-ND=改変禁止のため、説明文の加工転載や音声の組み込みはしない)

### 4.2 参照フロー

```
検出結果の headword
  → dictionary_entries を検索
     ├ ヒット: DBの値で表示を上書き + 「✓ 検証済み」バッジ + audio_path の音声を再生
     └ ミス: AIの値をそのまま表示 + 「AI生成・未検証」ラベル(小さく) + TTSで音声生成
         └ AI生成カードは source='ai' として dictionary_entries にも保存(次回から高速)
```

- 検証済みバッジ/未検証ラベルは**スキャンチップとカード詳細の両方**に表示
- ユーザーが誤りを見つけて訂正できる「報告・修正」ボタンをカード詳細に置く(自分のカードのみ修正可、共有 `words` テーブルの所有者チェックは docs/design/03 §1 の修正を前提とする)

### 4.3 音声の事前生成とキャッシュ

- 検証済み語: 別途バッチでTTS音声を一括生成し Storage に保存、`audio_path` に記録(バッチはClaude Code側で実行予定。アプリ側は audio_path があれば使うだけ)
- 未検証語: 初回タップ時にTTS生成 → `(言語, テキスト, 声)` キーで Storage にキャッシュ(既存の tts キャッシュ機構を Storage バックエンドで確認・整備)
- 声はアプリ全体で1つに統一(単語とネイティブの声を結びつけて記憶させるため)

## 5. キャッチ(保存)フロー

1. スキャンチップの「📷キャッチ」またはシャッターボタン → その物体を対象に保存開始
2. 切り抜き処理(現行の @imgly/background-removal によるブラウザ内処理を維持。API費用ゼロ)。**プリフェッチ済みの詳細カードがあればAI呼び出しなしで即カード完成**
3. 詳細確認 → 図鑑着地の一連の体験(§5.1)
4. 保存直後のシート(すべて任意・スキップ可): 自撮り追加 / 一言感想(その時何をしていた?どんな気持ち?) / 感情絵文字1タップ
5. 自動記録: 撮影日時・位置情報(EXIF/GPSは共有時に必ず除去)
6. 図鑑・今日のアルバム・地図に反映(既存機能)

### 5.1 キャッチ体験のシーケンス(機能要件)

視覚デザイン・色・アニメーションの具体的な作り込みはLovableのデザイン判断に任せる。ここでは**何がどの順で起きるか**だけを定義する。

1. **切り抜き完成の瞬間**: 撮った写真から対象物だけがステッカーのように分離される。ユーザーが「モノが自分のものになった」と感じる最初の山場
2. **詳細確認**: 切り抜きが画面上部に鎮座し、その下に単語の詳細(意味・発音・出典バッジ)が表示される。ユーザーはここで内容を確認・音声を再生できる
3. **保存確定**: ユーザーが保存を確定すると、切り抜きが**ぱっと浮き上がり、図鑑(ナビゲーションの図鑑アイコン)に吸い込まれてドンと着地する**。着地の瞬間に視覚・触覚(バイブ)・音の3つを同期させ、「図鑑に1体増えた」という手応えを作る
4. **着地後**: 図鑑アイコンにバッジ(+1)が付き、タップするといま入ったカードが図鑑の中で光っている

原則: 汎用的な紙吹雪・パーティクルの流用は禁止。演出は「収集物が自分の図鑑に物理的に収まる」という物語に沿ったものにする。

### 5.2 入力キャッチ(文字・音声)とフレーズカード

写真に撮れない出会い(授業で習った、店員に言われた、聞こえた)のための入口。

- スキャン画面にマイクボタンとキーボードボタンを併設
- **耳キャッチ**: マイクを押して聞こえたフレーズを自分で復唱 → Web Speech API(zh-TW)で文字起こし → 認識結果を確認・修正 → カード化。復唱自体が発音練習を兼ねる
- **文字キャッチ**: テキスト入力 → 辞書DB照合 → カード化(授業中など音を出せない場面用)
- **フレーズカード**(entry_type='phrase')の構造: 表=シーン(どこで・誰が・自分に何と言った/言いたかった)、裏=フレーズ+意味+**返し方の例**(「請稍等」→「好、謝謝」)。写真の代わりにシーン情報が記憶のインデックスになる。任意でその場の写真(店の外観など)を添付可
- フレーズカードの復習はロールプレイ形式: 相手役の音声を再生 → 自分が声で返す(§6の録音フローを流用)

### 5.3 ゴーストカードと再会演出(図鑑の「見つけた/捕まえた」)

- 入力キャッチで作られたカード(写真なし)は、図鑑で**ゴースト状態**として表示(写真ありカードと一目で区別できる見た目。既存の silhouette_emoji 列も活用可)。SRS復習には通常通り出る
- **仮画像**: 復習時にシルエットだけだと記憶の手がかりがなくストレスになるため、ゴーストカードには自動で仮画像を付ける
  - 取得順: ①フリーライセンス画像API(Openverse / Wikimedia Commons。出典クレジットを画像下に小さく表示)→ ②ヒットしない場合はAI生成イラスト
  - 任意のWebサイトからの画像の無断転載(スクレイピング)はライセンス上しない
  - 仮画像は写真と明確に区別できる見た目にする(「これはまだ仮」と分かることが、実物に会いたい動機=再会の喜びを守る)
- スキャン実行時、検出結果をユーザーのゴーストカードと照合し、一致したドットを**金色**で表示(§3.1b)
- 金色ドットをタップ →「◯日前に調べた『月台』だ!」の再会表示 → その場でキャッチ(撮影)すると仮画像が自分の写真に差し替わり、図鑑のエントリが完成する
- SRS上、再会キャッチは「現実世界での検索練習成功」として最高評価で記録し、次回復習間隔を大きく延長する
- 再会はキャッチ無料枠を消費しない(既にカードは存在するため。写真の追加だけ)
- 図鑑の統計は「見つけた数(ゴースト含む)」と「捕まえた数(写真あり)」を分けて表示する

## 6. 復習の作り替え(スピーキングoutput型)

### 基本フロー(1カード)
1. **自分の撮った写真**が表示される(場所・日時つき)。単語は表示しない
2. 問い: 「この時のことを、この単語を使って一文で話してみて」
3. マイクボタンで録音 → 音声認識でテキスト化(認識ミスは手入力で修正可能)
   - 音声認識はまず **Web Speech API**(ブラウザ内蔵の無料音声認識。zh-TW対応)を使い、非対応ブラウザでは録音→AIに文字起こしさせるフォールバック
4. **思い出せない場合**: 「ヒント」ボタン → 単語+発音が表示・再生される。SRS上は「忘れていた」として記録
5. 送信 → AIフィードバック画面:
   - 自分の発話文(修正可能)
   - 文法・自然さの添削(何が不自然でなぜかを日本語で)
   - **型の提示**: この単語をネイティブが使う頻出チャンク/コロケーション1つ+品詞色分けの語順パーツ表示(S/V/O/修飾で色分け)
   - ネイティブの気持ち・状況の解説(1〜2文)
   - お手本回答+別の言い方1つ
6. 「型を使ってもう一度言う」ボタン(任意の2周目)
7. 自分の録音を聞き直せる。**動画撮影はオプション**(設定でON時のみインカメ録画)

### ワードツリー(派生語の時間差解禁)

意味的干渉(意味が近い語を同時に学ぶと記憶の定着が悪化する現象)を避けるため、派生語は**同時に見せず、復習1回につき1枝ずつ解禁**する。

- キャッチ直後のカード表示は「単語+発音+意味+写真」のみ。プリフェッチ済みの詳細(コロケーション等)は**枝のストック**として非表示で保持
- 復習のたびに枝を1本解禁し、「型の提示」として使う(復習1回目:「吃芒果」、2回目:「芒果冰」…)
- 枝の順番: **つながり語**(コロケーション・よく一緒に使う語)を先に3〜4本 → **なかま語**(類義語・同カテゴリ語)はアンカーが安定した後(目安: 復習5回目以降)に解禁。意味的干渉を起こすのはなかま語のため
- カード詳細画面は「全情報の一覧」ではなく、中心=自分の写真、周囲=解禁済みの枝が放射状に生える**木のビュー**。未解禁の枝はロック表示(復習を続ける動機になる)
- 枝は独立カードにしない。ただし現実でその枝の実物をスキャン/キャッチしたら独立した木に昇格し、元の木とのリンクを保持する(図鑑に入るのは現実で出会った言葉だけ、の原則)

### SRSへの接続
- ヒント未使用で言えた=成功、ヒント使用=失念、スキップ=失敗として既存SRSスケジューラに渡す
- 何度も失敗するカード: 解説の再生成を提案 → それでもダメなら「もう一度実物を撮りに行こう」クエストを提案
- 既存の4択は「ライトモード」(声を出せない場所用)として設定に残す(§10 未決)

## 7. AI呼び出し設計と高速化

### プロバイダとモデル(2026年7月時点の確定版)

`.env` 設定(コード変更不要、既存の ai-provider.server.ts が対応済み):
```
AI_PROVIDER=google
GEMINI_API_KEY=<Secretsフォームで安全に登録。チャットには貼らない>
AI_MODEL_FAST=gemini-3.1-flash-lite     # スキャン検出・誤答生成・文字起こし用
AI_MODEL_RICH=gemini-3.1-flash-lite     # カード詳細・添削・お手本生成用(不満が出たら 3.5-flash に上げる)
```

**モデル選定の根拠(2026-07-04 時点、公式価格ページで確認)**:
- 検出係・解説係とも **Gemini 3.1 Flash-Lite** に統一。理由: 2026年5月に正式版化(preview卒業)して安定。有料枠でも入力 $0.25/出力 $1.50 per 1M tokens と最安クラス。無料枠対象
- コードに元々あった `gemini-3-flash-preview` は世代が古くpreview(名前が変わるリスク)。入力コストも Flash-Lite の倍
- 解説の品質が不足する場合のみ `AI_MODEL_RICH` を `gemini-3.5-flash`(入力 $1.50/出力 $9.00、最新最上位Flash)に上げる。1行変えるだけで切替可能
- 2.0系(2.0 Flash / Flash-Lite)は2026年6月1日に**完全終了**。使用不可
- モデル名・料金・無料枠は変動するため、実装時に https://ai.google.dev/gemini-api/docs/pricing で必ず最新確認

**無料枠→有料枠の移行設計**:
- MVP開発中は無料枠(Flash系のみ利用可、Pro系は不可)。無料枠は入出力トークン課金ゼロだが RPM/RPD(1分/1日あたりの回数上限)制限あり。開発テストで RPD を使い切ることがあるので、Flash-Lite(最も枠が多い)を主役にする
- ローンチ時に有料枠へ: 課金を有効化すると①回数制限が大幅緩和 ②入出力データがGoogleの学習に使われなくなる(プライバシー保護)。有料化後もモデル名は同じでよく、`.env`の変更は不要
- 有料化後の追加節約策: **コンテキストキャッシュ**(同じ長い指示文を繰り返し使う場合に9割引きになる仕組み)と **Batch API**(急がない一括処理が5割引き)。辞書の事前TTS生成やAI解説の一括先行生成に使える

### TTS(音声読み上げ)
- まず Gemini系TTS または Lovableゲートウェイ既存設定で開始。品質不満なら Google Cloud TTS の zh-TW 音声へ切替(TTS_BASE_URL 等の既存の差し替え口を使う)
- 検証済み語の音声は事前一括生成(Batch API利用可)。生成済み音声はStorageにキャッシュし再生成しない

### 速度の4原則(必ず実装)
1. **画像はクライアントで縮小してから送る**(長辺1024px・JPEG品質0.7)。※現行コードは768px縮小済みで実装確認済み
2. **検出コールはJSONのみ返させ、思考モード/長文説明を無効化**(maxOutputTokensも小さく)
3. **詳細生成はタップ時にプリフェッチ**、タップされない物体には使わない
4. **音声と辞書はキャッシュ優先**(Storage キャッシュ → なければ生成)

### 計測(受入の前提)
`scan_events` に `detect_ms`(スキャン→ドット表示)と `tap_to_audio_ms`(タップ→音声再生開始)を記録し、設定画面の開発者欄で直近20回の中央値を見られるようにする。

## 8. 実装フェーズ分割(Lovableへ1つずつ送る)

### Phase 0: プロバイダ切替
> .env の AI_PROVIDER を google に切り替え、GEMINI_API_KEY を設定して全AI機能が動くことを確認して。AI_MODEL_FAST / AI_MODEL_RICH を環境変数で上書きできることも確認。TTSの現在の経路(どのAPIで生成しているか、キャッシュはStorageに保存されているか)を調査して報告して。コードの変更はしないで。
> ※切り抜きは @imgly/background-removal によるブラウザ内処理と確認済み(API費用ゼロ)。この方式を維持する。

**受入基準**: 既存の撮影→候補→カード生成がGemini直結で動く。TTS経路の報告が得られる。

### Phase 1: 辞書テーブルとスキャンログ
> §4.1 の `dictionary_entries` テーブルと §3.7 の `scan_events` テーブルをマイグレーションで追加して。RLS: dictionary_entries は全ユーザー読み取り可・書き込みはサーバー関数経由のみ。scan_events は本人のみ読み書き可。CSVで dictionary_entries に一括投入できる簡易管理ページ(自分のアカウントのみアクセス可)も作って。

**受入基準**: テーブルが作成され、CSV投入→検索が動く。

### Phase 2: スキャンモード画面
> §3.1〜3.2 の仕様でスキャンモードを新規実装して。カメラプレビュー+スキャンボタン→1フレームをAI検出(§3.2のJSON仕様、AI_MODEL_FAST=gemini-3.1-flash-lite を使用)→正規化座標にドット表示(モノと文字が見分けられる区別、スタイルはあなたのデザイン判断で)→タップで単語チップ表示+音声再生。辞書DBヒット時は「✓検証済み」バッジとDB値を優先表示、ミス時は「AI生成」ラベル。confidence<0.75 は候補チップ(§3.6)。検出前に画像を長辺1024pxに縮小。scan_events への記録と detect_ms / tap_to_audio_ms の計測も実装。

**受入基準**: スキャン→ドット表示が中央値2.5秒以内、タップ→音声再生が1秒以内(検証済み語)。誤認識時に候補チップから選び直せる。

### Phase 3: プリフェッチとキャッチ統合
> §3.3 のプリフェッチ(タップ時に generateCard をバックグラウンド開始、セッション内キャッシュ)と、§5 のキャッチフロー(スキャンチップからのキャッチ、プリフェッチ済みカードの再利用、§5.1のキャッチ体験シーケンス、任意の自撮り・一言感想シート)を実装して。§5.1 は機能要件なので、演出の具体的な質感・動き・音はあなたのデザイン判断で最高のものにして。タップされていない物体の詳細生成は絶対に行わないこと。

**受入基準**: タップ→3秒以上眺めてから「詳しく」を開くとカードが既に表示される。キャッチ時の追加AI呼び出しが(プリフェッチ済みなら)ゼロ。

### Phase 4: 復習リデザイン
> §6 の仕様で復習をスピーキングoutput型に作り替えて。写真表示→録音(Web Speech API、修正可能)→ヒントボタン(SRS失念記録)→AIフィードバック(添削・型の提示・品詞色分け・お手本)。既存の4択はライトモードとして設定でON/OFF。動画撮影は設定でONのときだけ。

**受入基準**: 1カードの復習が録音→フィードバックまで途切れず完了する。ヒント使用がSRSに反映される。

### Phase 5: 磨き込み
> スキャン解析中の待ち時間演出、キャッチ着地体験の質感向上(§5.1の意図に沿って)、「+細かく」階層ドット(§3.5)、速度計測値の開発者向け表示を実装して。演出のデザインはあなたの判断で。

**受入基準**: §9 の数値目標をすべて満たす。

### Phase 6: 入力キャッチ・ゴースト再会・ワードツリー
> §5.2 の入力キャッチ(文字・音声復唱、フレーズカード)、§3.1b のドット状態システム(新発見/再会=金色/取得済み/通常の4状態。金色以外の見た目はあなたのデザイン判断で)、§5.3 のゴーストカード(仮画像の自動取得: フリーライセンス画像API→なければAI生成イラスト、出典表示付き。再会キャッチでの写真差し替えとSRS最高評価、見つけた数/捕まえた数の分離表示)、§6 のワードツリー(枝の時間差解禁、カード詳細の木ビュー、未解禁枝のロック表示)を実装して。

**受入基準**: 文字入力→ゴースト作成(仮画像付き)→スキャンで同じ単語を検出→金色ドット→キャッチで写真差し替え、の一連が動く。初スキャンの未知語ドットが「新発見」状態で表示される。復習2回目で2本目の枝が解禁される。
