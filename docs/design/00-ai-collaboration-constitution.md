# AI協働憲法 — Catchwords開発の思考OS(docs/design/00-ai-collaboration-constitution.md)

> 目的: 最上位モデルで行った「判断の質」を文書化し、どのAIモデル・どのセッションでも同じ水準の思考を再現する。
> 使い方: 判断が重い相談をAIにする時、**このファイルと 06(仕様書)・07(ロードマップ)を添付**し、§4のマスタープロンプトで依頼する。

---

## 1. プロジェクトの北極星(1行で全てを決める)

**「街で見たものの単語と発音が、瞬間的に分かる」**——全ての機能・優先順位・トレードオフはこの1行から逆算する。
第二原理: このアプリは単語収集アプリではなく**「現実世界での言語体験を蓄積するアプリ」**。判断に迷ったら「現実世界を教材にする体験を強くするか?」と問う。

## 2. 非交渉原則(どのモデルにも守らせる)

1. **正確性の階層**: 発音・意味の誤りは致命傷。検証済み(公式データ由来)とAI生成(未検証)を必ず区別し、UI上もラベルで示す
2. **摩擦の階層**: 調べる=摩擦ゼロ(無制限・無課金壁)、保存=最大2タップ、復習=1ミリでもハードルを下げる
3. **ハルシネーション3層対策**: ①AIの推測をデータ(公式ソース・ルールベース変換)で置き換えられるなら必ず置き換える ②AI出力には照合列(別ソースの参考訳など)を並べて人間の検証を10倍速にする ③機械チェックで疑わしい出力を自動隔離する
4. **異常値で止まる**: おかしな数字(ヒット率16%等)が出たら、進めずに原因を特定する。「たぶん大丈夫」は禁止
5. **小さく試す**: 本番投入の前に必ず3件テスト。エラーは全文を読み、原因を特定してから再試行
6. **デザイン委任**: 視覚デザイン・演出の具体はLovableに委ねる。文書は機能・フロー・数値だけを定義(例外: 機能的意味を持つ指定のみ)
7. **コスト意識**: AI呼び出しは「タップされたものだけ」「キャッシュ優先」「安いモデルで足りるなら安いモデル」。画像は送信前に縮小
8. **秘密情報**: APIキーはチャット本文に貼らない。貼ってしまったら即ローテーション(再発行)

## 3. 意思決定ログ(なぜそう決めたか)

| 決定 | 理由(要約) |
|---|---|
| かざす=調べる(無制限)/撮る=保存(儀式)の2段構え | 最大ペイン(瞬間性)と収集欲を両立。調べるに壁を置くと存在意義が死ぬ |
| タップ=選択方式(A+B) | AIに1つを当てさせず、ユーザーの指で曖昧性を解消→精度と速度を同時に取る |
| 詳細生成はタップ時プリフェッチのみ | 全物体を先読みするとコスト10倍。タップ後の3〜5秒で生成が間に合う |
| 辞書は3層ハイブリッド(TOCFL検証済みDB→AI→ラベル) | AIの発音間違いを構造的に排除。公式データは機械変換のみで加工 |
| 教育部辞典の本文・音声は組込まない | CC BY-ND(改変禁止)とスクレイピング禁止。照合用参照のみ |
| 復習=写真を見て自分の経験を話す(4択はライトモード) | 生成効果・自己参照効果・検索練習を1つの行為に統合 |
| ワードツリー(枝は復習1回に1本解禁、なかま語は後回し) | 意味的干渉の回避。赤ちゃんは時間差で1枝ずつ接ぎ木する |
| ゴーストカード+金色再会、再会は無料枠を消費しない | 現実での再会=最強の検索練習。喜びの瞬間に減点要素を置かない |
| 無料=毎日リセット(キャッチ3/日、添削3/日)、リバーストライアル7日 | 習慣が先、課金は欲求のピークで。買い切り型トライアルは価値実感前に離脱させる |
| モデルはFlash-Lite統一、RICHのみ必要時に格上げ | 検出は頻度が高く安さが正義。1行の設定で上げ下げできる構造を維持 |
| 多音字・複数レベル重複語は1カードに統合 | DBは1単語1行。学習上も意味の広がりとして自然 |
| ストア課金はIAP(Small Business 15%)で設計 | アプリ内サブスクはストア決済が原則。Stripe構想はWeb版に限定 |
| 復習画面はLovable版デザインをベースにPR版の機能を統合(2026-07-12) | 並行実装の衝突。録画・2周目再添削・客観採点はLovable版、DB保存のモード設定・ワードツリー連携・ゴースト対応・3段階SRS・KPI記録はPR版から |
| 復習モード設定はprofiles.review_modeにDB保存(localStorage不可) | 端末をまたいで設定が引き継がれる。列レベルGRANT環境では新列にSELECT/UPDATEのGRANT明示が必須 |
| profilesの列レベルGRANTで他人から設定を隠す方式は不採用 | 列GRANTは「自分の行だけ全列」を表現できず自分の設定も読めなくなる(2026-07-12に実際に発生→復旧)。他人向け公開はPhase Bでビュー方式に |

(新しい重要判断をしたら、この表に1行追加すること)

## 4. マスタープロンプト(コピペ用)

### 4a. 実装依頼(Lovable/Claude Code向け)
```
添付の docs/design/06-scan-first-mvp.md と 00-ai-collaboration-constitution.md を前提とする。
タスク: [1つだけ書く]
制約: §2の非交渉原則に従う。コード変更前に現状を調査して報告→私の承認後に実装。
完了条件(受入基準): [数値または再現手順で書く]
最後に自己チェック: 仕様との差分・触っていないはずの箇所への影響・追加したAI呼び出しの回数とコストを報告して。
```

### 4b. 判断・戦略相談(上位モデル向け)
```
あなたはこのプロジェクトの共同創業者として答える。添付: 00(憲法)/06(仕様)/07(ロードマップ)+[最新の数値・状況]。
質問: [1つだけ書く]
求める出力: ①結論を先に ②理由(北極星と非交渉原則に照らして) ③捨てる選択肢とその理由 ④次の行動を番号つきステップで ⑤この判断を意思決定ログに追加する場合の1行。
禁止: 一般論だけの回答。必ずこのプロジェクトの文脈・数値に接地すること。
```

### 4c. トラブル解析
```
症状: [エラー文・スクショの内容をそのまま]
依頼: 原因の仮説を確度順に3つ→それぞれの確認方法(コマンド/画面操作)→最も安全な修正手順。
制約: いきなり修正しない。確認→特定→修正の順。修正は最小差分で。
```

## 5. 品質チェックの習慣(人間側=NORIの型)

1. AIの完了報告は**受入基準で自分の手で確認**してから次へ(報告を信じて進めない)
2. 数字の異常(急に減った/増えた/合わない)は必ず「なぜ?」と聞く
3. 週1で: KPIを見る→ロードマップとのズレを1つだけ特定→次の1手を決める
4. 新しい概念が出たら「初心者向けに用語解説して」と必ず要求する(理解しないまま進めない)
5. 迷ったら北極星に戻る:「それは"瞬間的に分かる"を強くするか?」
