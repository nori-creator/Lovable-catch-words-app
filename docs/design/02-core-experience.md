# 02. コア体験の詳細仕様

各セクションは「なぜ」→「仕様」の順。仕様ブロックはそのまま Lovable / Claude Code への指示に使える粒度で書いている。
DBの変更は [04 §6](./04-architecture.md) のSQLスケッチに集約。

## 1. 画面構成(確認)

5タブ構成は現状のままで良い: **ホーム(今日のアルバム) / 図鑑 / [撮る] / 復習 / 設定**。
地図は図鑑内のビュー切替(グリッド⇄地図)にすると、タブを増やさず「場所から思い出す」導線を保てる。
ソーシャル(feed)はホーム上部のストーリーズ風の帯に格納し、独立タブにしない
(タブを増やすほど毎日の動線が薄まる。あなたの「なるべくシンプルに」に一致)。

## 2. 撮影フロー: 2段階AI解析(最重要)

### なぜ

ペイン①の本質は「知りたい瞬間に答えが出ない」こと。いまの実装は
候補5件の生成(`suggestWords`)→ユーザー選択→カード生成(`generateCard`)が**直列**で、
意味が分かるまで体感10秒を超えうる。人は3秒待たされると「調べる」に戻ってしまう。

### 仕様: 速報パス(fast pass)+ 詳細パス(enrich pass)

```
シャッター
  ├─(並列)クライアント側で切り抜き開始(@imgly/background-removal は現状どおり)
  └─(並列)速報AIコール 1回:
        入力: 縮小画像(長辺768px程度に圧縮してアップロード時間も短縮)
        出力: 候補3件 { headword, meaning_ja, pinyin, zhuyin, confidence }
        → 第1候補を2〜3秒で画面に大きく表示 + TTS再生(キャッシュヒット時は即時)
        → 「これじゃない」なら残り候補 or 手入力
ユーザーが候補を確定(または3秒間無操作で第1候補を自動確定)
  └─ 詳細パス(バックグラウンド):
        generateCard 相当(例文・コロケーション・語源・extras 全部)
        → 完了したらカードの下部に「解説が届いた」演出でぬるっと展開
保存はユーザー確定の瞬間に行い、extras は後から UPDATE で埋める
```

- レイテンシ予算: シャッター→意味+音声 **3秒以内**、→詳細カード完成 **15秒以内**、→保存完了は非同期。
- 速報パスの出力はキャッシュを最大活用: `words` テーブルに同じ headword があれば **AIを呼ばず既存カードを即返す**(このために現状の words 共有設計は正しい)。街の頻出語(「停」「營業中」等)はすぐ全ユーザーでキャッシュが温まる。
- 「解析中」の待ち時間は、切り抜きが浮かび上がる演出・進捗コピー(「ネイティブに聞いています…」)で報酬感に変換する(あなたのエフェクト要望と同じ方向)。
- **設定に「クイックモード」**: 速報パスだけで保存し、詳細パスは図鑑で開いた時に遅延生成。「意味と発音だけ欲しい人」向け(あなたの要望どおり)であり、無料枠のAIコスト削減にもなる。

### 発展: 撮影レス入力

- 音声入力: マイク長押し→STT→headword 確定→同じ詳細パス。
- 文字入力: 既存の text モードを維持。**文字入力で覚えた語も図鑑に「未遭遇」マークで入れる**→街で実物を撮った瞬間「遭遇済み」に変わる。あなたの「授業で習った単語に街で出会うと嬉しい」をデータ構造で表現する(→ §4 再遭遇)。

## 3. 復習(SRS)セッションの再設計

### 3.1 段階式復習: 認識 → 聴解 → 産出

SM-2 の `repetitions`(既存カラム)をそのままモード昇格に使う。
「4択だけでは産出に至らない」(ペイン③⑨)への解。

| repetitions | モード | 出題形式 |
|---|---|---|
| 0–1 | 認識 | 自分の写真+単語を見て、意味を4択 |
| 2–3 | 聴解 | **音声だけ**が流れる(文字なし)→意味を4択。写真は回答後に表示 |
| 4–5 | 逆産出 | 意味(日本語)+写真→正しい単語を4択(繁体字の見分け) |
| 6+ | 産出 | 写真だけ→**声に出して言う**(Web Speech APIで判定、§3.5)。フォールバックはタイプ入力 |

- 各モードで正解時のSRSスコアは同じ扱いにする(モードはあくまで「難易度の階段」)。
- 産出モード非対応環境(STTなし)では逆産出に留める。graceful degradation は既存 `PronunciationPanel` の方針を踏襲。

### 3.2 音声の自動再生と TTS キャッシュ

- 復習カード表示と同時にネイティブ音声を自動再生(あなたの要望)。連打防止に再再生ボタンも置く。
- 前提として **TTSキャッシュ** が必須。現状 `synthesizeSpeech`(`src/lib/tts.functions.ts`)は毎回 gpt-4o-mini-tts を呼び base64 を返しており、同じ単語で何度も課金+遅延している。

仕様:
1. Storage バケット `tts` を作成し、キーは `{language}/{voice}/{sha256(text)}.mp3`
2. `synthesizeSpeech` は最初に Storage を見て、あればsigned URLを返す。なければ生成→Storageに保存→URL返却
3. `words` に `audio_path` 列を持たせ、カード保存時(詳細パス内)に headword + 例文のTTSを先回り生成しておく → 復習・図鑑では**AI呼び出しゼロで即再生**

### 3.3 4択の誤答(distractor)は事前生成+自家製に

現状の問題: `getDueReviews`(`src/lib/reviews.functions.ts:105`)が復習を開くたびに
カード1枚あたり Maker/Checker で AI 2往復以上 × 最大10枚を**直列**実行。復習開始が数十秒級になり、コストも毎回かかる。

仕様(優先順):
1. **自家製誤答(AI不要・最優先)**: 自分が収集済みの「同カテゴリの別の単語」の meaning_ja から3つ選ぶ。
   - 教育的にも最良: 自分が知っている語同士の弁別こそが本当の混同ポイント(「蘋果」と「鳳梨」を間違える人に「別の物体」という誤答は無意味)。
   - 同カテゴリが3語未満の間だけ次のフォールバックへ。
2. **事前生成キャッシュ**: カード保存時(詳細パス)に Maker/Checker を1回だけ回し、`review_choices` テーブル(→ [04 §6](./04-architecture.md))に難易度3段階×各3個を保存。復習時はここから引くだけ。
3. **毎回違う4択**というあなたの要望は「プール(自家製+事前生成で6〜12個)からランダムに3個引く」ことで満たす。毎回のAI生成は不要。
4. 難易度設定(設定画面: やさしい/ふつう/いじわる)はプールのどの層から引くかの切替。

### 3.4 リーチ(何度も忘れる語)の自動ケア

ペイン⑧「作り替えの面倒」への解。SRSで検知して**アプリ側から**提案する。

仕様:
- 条件: 直近の `review_history` で3回連続 `correct=false`、または lapse(repetitions リセット)が累計3回
- 発火する提案(復習セッション終了画面に表示):
  1. 「ニーモニックを作り直す」→ `mnemonic` と `trivia` だけ再生成(項目単位の再生成、§6.2)
  2. 「もう一度実物を撮りに行く」→ 明日のデイリークエストにこの単語を自動追加(既存 `daily_quests` を流用)+ 地図に「ここで撮った」ピンを強調
  3. 「例文を自分の生活で作る」→ AIがユーザーのキャプション・場所を織り込んだパーソナル例文を生成
- 再撮影が完了したら §4 の再遭遇ボーナスで SRS を仕切り直す。

### 3.5 発音チェックの段階導入

- **Phase 1(いま)**: 既存 `PronunciationPanel` の Web Speech API 判定を産出モード(§3.1)に組み込む。判定は「認識テキスト==headword」の粗い一致+ピンイン比較。無料。
- **Phase 2**: サーバ側STT(音声をアップロード→STTモデル→ピンイン変換して比較、声調の正誤を4声単位でフィードバック)。Pro機能の柱。
- **Phase 3**: 音素・声調レベルのスコアリング(専用の発音評価API、zh-TW対応のもの)+口形・波形の可視化。サブスク「発音コーチ」。
- モンスターボール演出(発音成功でキャッチ完了)は Phase 1 の技術でも実装可能: capture 確定画面に「発音してゲット」ボタン→成功で追加XP+専用エフェクト。**通常保存も残す**(騒がしい場所で発音できないと保存すらできない、にしない)。

## 4. 再遭遇ループ(新規・コアループ昇格)

### なぜ

あなたのメモに繰り返し出てくる「何回も見かけるのに覚えられない」「忘れて撮るたびレベルアップ」「何回も撮ったランキング」は、
バラバラの機能ではなく**1つのループ**として設計すべき。現実世界で同じ物に再会した瞬間の想起テストは、
アプリ内SRSより強い記憶イベント(文脈・感情つきの検索練習)になる。

### 仕様

1. capture の速報パスで、確定した headword が既に自分の図鑑にある場合、保存フローを分岐:
   - 通常の「新規ゲット」ではなく **「再遭遇!」モード**に入る
   - まず意味を隠して「これ、なんだっけ?」→思い出してからタップで答え合わせ(2秒の想起練習)
   - 正解を見た後、写真は既存ステッカーの `encounters` に追記(→ [04 §6](./04-architecture.md))。図鑑のカード詳細に遭遇履歴(日時・場所・写真)がタイムライン表示される
2. SRSへの反映: 再遭遇時に該当 review を「その場で1回分消化」扱いにし(思い出せた=correct)、`due_at` を前倒しでなく通常のSM-2更新。思い出せなかった場合は lapse として扱う
3. 遭遇カウントで図鑑カードに「×3」バッジ。**遭遇回数ランキング**(あなたの案)は図鑑のソート順として実装(「よく出会う順」)
4. 文字入力で先に登録した語(§2)は `first_encounter_at` が null → 街で初めて撮った瞬間に「初遭遇!」演出。教室の知識が現実と接続する瞬間を最大限に祝う

## 5. 図鑑 × 記憶の可視化(ペイン④の完成形)

- 図鑑グリッドの各ステッカーに、`getOverallMemoryStats` と同じ保持率モデルで**記憶状態の色枠**を付ける:
  緑(80%+)/黄(50–80%)/赤(50%未満)/グレー(未復習)
- カテゴリヘッダに **コンプ率「果物 8/20」**。分母はカテゴリ×TOCFLレベル別の推奨語彙リスト(静的テーブル、→ [03 §4](./03-overlooked.md) の語彙表整備と同時に作る)。未収集分は `silhouette_emoji`(既存カラム)でシルエット表示 → ポケモン図鑑の「まだ見ぬ枠」による収集欲
- 図鑑上部にビュー切替: グリッド / 地図 / 日付。日付検索(あなたの要望)はここに入れる
- タップ→詳細(既存 `dex.$stickerId`)に、遭遇タイムライン(§4)と忘却曲線(既存 `ForgettingCurveChart`)を統合

## 6. 単語カード(WordCard)の強化

### 6.1 品詞色分けチャンク例文(あなたの要望の実装仕様)

例文を文字列でなく**チャンク配列**で生成させる:

```json
{
  "example_chunks": [
    { "text": "我", "role": "S" },
    { "text": "吃", "role": "V" },
    { "text": "蘋果", "role": "O", "is_headword": true },
    { "text": "每天早上", "role": "ADV" }
  ]
}
```

- role → 色のマッピングを固定(S=青 / V=赤 / O=緑 / ADV=橙 / その他=グレー)。凡例をカード下部に常設
- 複数品詞がある語は品詞ごとに例文チャンクを1つずつ生成(語法解説も品詞別、既存の `usage_note` を品詞別配列に拡張)
- 既存データとの互換: `extras.examples_extra` は残し、新フィールド `example_chunks` を追加(jsonbなのでマイグレーション不要)

### 6.2 項目単位の再生成と編集(ペイン⑧)

- カード詳細の各セクション(語源/ニーモニック/例文/コロケーション…)に「🔄 作り直す」ボタン
- 再生成プロンプトには「前回の内容」と「ユーザーの一言(任意入力: 例『もっとダジャレっぽく』)」を渡す
- 実装は既存 `updateWordExtras` を項目パッチで呼ぶだけだが、**所有権の修正が先**([03 §1](./03-overlooked.md)): words を共有したまま個人の上書きを許すと他人のカードが変わる。`user_word_overrides` テーブル(→ [04 §6](./04-architecture.md))に個人差分を保存し、表示時にマージする
- 項目の並べ替え・表示/非表示(あなたの要望)は `profiles` に `card_layout` jsonb を追加してドラッグで編集

### 6.3 「実際に使われている」リンク

AIにURLを直接生成させない(存在しないリンクを作る)。代わりに検索URLを組み立てる:
- 動画: YouGlish系サービスの検索URL(`headword` を埋め込む)
- 用例: 教育部辞書(萌典)のエントリURL、Google ニュース検索 `"{headword}" site:.tw`
- 将来: 自前で例文動画DBを持つのは Later。まずは外部リンクで十分価値が出る

### 6.4 頻度による視覚差(あなたの要望)

- 語彙表整備([03 §4](./03-overlooked.md))後、頻度ランクをカードに反映: 高頻度語はヘッドワードを大きく・「日常必須」バッジ、低頻度語は「レア語」バッジ(収集ゲーム的にはレアの方が嬉しい、という逆転の演出も可)

## 7. 位置リマインダー(「同じ場所を通ったらテスト」)

技術制約が本質: **PWAはバックグラウンドで位置を取れない**。段階設計にする。

- **Phase 1(PWA)**: アプリ起動時と地図表示時に現在地を1回取得→自分のステッカーで「半径150m以内 & due が近い(±3日)」を照合→あれば「📍この近くで撮った『珍珠奶茶』、まだ覚えてる?」というその場クイズカードをホーム最上部に出す
- **Phase 2(PWA+通知)**: Web Push が使える環境では、SRS due 通知に「場所ヒント」を混ぜる(「駅の近くで撮ったあの単語が忘れかけです」)。位置での発火はまだ不可
- **Phase 3(Capacitor)**: ネイティブのジオフェンス API で本物の「通りかかったら通知」。バッテリー配慮で登録ジオフェンスは「due が7日以内のステッカーの場所、最大20件」に限定
- プライバシー: 位置照合は端末内 or 自分の行だけを読むサーバ関数で完結させ、位置履歴の常時収集はしない([03 §2](./03-overlooked.md))

## 8. 週間リキャップとInstagram輸出(成長エンジン)

アプリ内SNS(feed/likes は実装済み)はコールドスタート問題が重い。初期は**既存SNSへ「作品」を輸出**して認知を取る方が効率的。

仕様:
- 毎週日曜、その週のステッカーからリキャップ画像を自動生成(canvas でクライアント生成: 切り抜きコラージュ+「今週 12語 ゲット・3日連続」+アプリロゴ)
- ホームに「今週のアルバムが完成!」カード→タップでプレビュー→OS共有シート(IGストーリーズ/LINE)
- 1日のアルバム(既存ホーム)にも同じ「画像として書き出す」ボタン
- 書き出し画像には位置情報を焼き込まない(地名は「台北市」レベルまで)。自撮りを含めるかは書き出し時に毎回トグル([03 §2](./03-overlooked.md))

## 9. ストリークとXPの一本化

現状XPは `stats.functions.ts` 内でその場計算(review スコア×2 など)されるだけで永続化されていない。ゲーム経済は一箇所に集める。

仕様:
- `xp_events` テーブル(→ [04 §6](./04-architecture.md))に全獲得を記録: capture +10 / 再遭遇 +15 / 復習正解 +score×2 / クエスト達成 +20 / 日記添削 +10 / 発音ゲット +5
- ストリーク定義: 「その日に capture または復習を1回以上」で継続。**フリーズ(月2回まで自動救済)**を最初から入れる — ストリーク切れは最大の離脱イベントなので、罰ではなく救済を設計する
- ホームのヘッダに 🔥ストリーク / 今日のXP / due数 を常設(毎日開く理由の集約点)
- レベルはXP閾値の単純テーブル。キャラ育成(あなたの構想)は Later でこの上に乗せる(XP基盤が先)

## 10. 日記ループの強化

既存: 今日のキャプチャから模範日記生成(`generateTodayJournal`)+ユーザー下書き添削(`correctMyJournal`)。良い骨格。

追加仕様:
1. **書けない人への足場(スキャフォールド)**: 白紙に書かせない。「今日の単語を使った3つの文型テンプレ(穴埋め)」をAIが提示→穴埋めするだけで日記が完成→それを添削へ。レベル設定(単語だけ/穴埋め/自由作文)
2. 添削結果の差分表示: correction を全文でなく**変更箇所ハイライト**(diff)で表示。feedback_ja の各指摘を該当箇所にひも付け
3. 「言いたいことを日本語で書く→AIが台湾華語のお手本+解説」モード(あなたの案)を第3タブとして追加
4. 日記から出た未知語はワンタップで図鑑に登録(文字入力capture の導線再利用、`first_encounter_at=null` で§4に接続)
5. 週間リキャップ(§8)に今週の日記のベスト1文を載せる
